Here is the breakdown of the criteria and the rationale for placing each suggested crate into the Core architecture:

### 1. The Core vs. Extension Decision Criteria

An **Extension Module** (using the `tool-sdk/`) is a service the AGI chooses to call to achieve a specific goal (e.g., search the web, send an email, run a utility).

A **Core Module** is a critical component that manages the internal operation, safety, compliance, or performance of the AGI itself. If a Core Module fails, the entire AGI system is compromised.

| Criteria | Core Module (Tightly Integrated) | Extension Module (Loosely Coupled) |
| :--- | :--- | :--- |
| **Necessity** | Essential for system function, safety, or compliance. | Optional for task completion; functional utility. |
| **Latency** | Requires sub-millisecond, low-latency access (often in-process). | Tolerates network latency (gRPC/HTTP call). |
| **Integration** | Requires deep, cross-cutting access to core state (KBs, Orchestrator). | Communicates solely via the `tools-service-rs` interface. |
| **Coupling** | High (must be developed and deployed with the main system). | Low (can be developed/deployed independently by third parties). |
